Prompt
Build a full-stack Contract Lifecycle Management (CLM) web application using React (frontend) + Node.js/Express (backend) + PostgreSQL (database). The app uses Google OAuth 2.0 for authentication and Google Drive API for document storage in a Google Shared Drive. I already have a Google Cloud project with OAuth credentials and the Drive API enabled.

AUTHENTICATION & USER MODEL

Users sign in exclusively via Google OAuth 2.0. Use Passport.js with the Google strategy.
On first login, create the user in the database. Pull their name, email, profile photo, and Google Workspace organizational unit / department from the Google Directory API (admin.googleapis.com) or the People API. Store the department on the user record — it is used for workflow routing.
There are four roles: Submitter (default for all authenticated users), Legal Reviewer, Designated Signer, and Admin. A user can hold multiple roles. Roles are assigned by an Admin through the admin panel.
Protect all routes with middleware that checks authentication. Protect admin routes with middleware that checks the Admin role.
Use express-session with a PostgreSQL session store (connect-pg-simple). Store the Google access token and refresh token so the app can make Drive API calls on behalf of the user.
Also configure a Google Service Account with domain-wide delegation. The service account is used for backend operations (AI screening, automated reminders, file management). Store its credentials securely in environment variables.

AI PRE-SCREENING (Feature #3 — AI Contract Intelligence & Risk Scoring)
When a contract is submitted, a background job triggers AI review using the Anthropic Claude API (or OpenAI — use whichever is available; make the provider configurable via environment variable).
The AI screening process:

Retrieve the document from Google Drive using the service account.
Extract text from the document (use pdf-parse for PDFs, mammoth for DOCX).
Send the extracted text to the LLM with a structured prompt that asks it to validate admin-configured criteria and return a JSON response.
Admin-Configurable Criteria — the admin panel has a "Screening Criteria" section where the admin defines validation rules. Seed with these defaults:
CriteriaDescriptionEnabled by DefaultCompany Legal NameCounterparty name in document matches the name entered in the formYesAddress VerificationAddress in document matches the address entered in the formYesRequired ClausesCheck for presence of: indemnification, termination, confidentiality (admin can add/remove clause types)YesDates Present & ValidEffective and expiration dates exist and are logically valid (expiration after effective)YesParties NamedNo blank placeholders like [PARTY NAME] or TBD in party fieldsYesSignature BlocksAll required signature blocks are presentYesContract ValueValue stated in document is present and within the authorized range for this contract typeYes

Risk Score: The AI returns a score: Low (0-1 issues), Medium (2-3 issues), High (4+ issues). Each criterion gets a pass/fail with an explanation.
Result Handling:

All pass (Low risk): Contract automatically advances to the next workflow stage (Legal Review).
Medium risk: Contract advances to Legal Review but the risk summary is flagged prominently on the contract detail page.
High risk: Contract is returned to the submitter with the AI's findings displayed as actionable feedback. Submitter must fix and resubmit.


Store the full AI screening result (JSON) in the database linked to the contract, including timestamp and the criteria version used.


WORKFLOW ENGINE
The workflow engine is the core of the app. Workflows are fully admin-configurable.
Workflow Determination: When a contract is submitted, the system determines which workflow to use based on three factors:

Direction — Buy-side or Sell-side
Department — from the submitter's Google Workspace profile
Contract Value Tier — the admin defines 4+ tiers with dollar thresholds (e.g., Tier 1: $0–$10K, Tier 2: $10K–$50K, Tier 3: $50K–$250K, Tier 4: $250K+). Tiers are admin-configurable with no hard limit on the number of tiers.

Workflow Definition: Each workflow is a sequence of stages. The admin creates workflows in the admin panel with:

Workflow name
Matching rules (direction + department + value tier range)
Ordered list of stages, where each stage has:

Stage name (e.g., "Legal Review", "Finance Review", "VP Approval")
Assigned role or specific user(s)
Can the reviewer send the contract back to the submitter? (yes/no toggle — yes by default)
Is this stage optional or required?
Designated signer for the signature stage (specific user, determined by department + tier)



Seed a default workflow:

Stage 1: AI Screening (automatic)
Stage 2: Legal Review (assigned to Legal Reviewer role)
Stage 3: Signature (assigned to Designated Signer — determined by department + value tier)
Stage 4: Awaiting Executed Upload (assigned to Submitter)

Workflow Behavior:

At each review stage, the reviewer sees the contract details, the attached document (embedded viewer or link to Drive), the AI screening results, all prior comments, and action buttons: Approve, Send Back for Edits, Add Comment.
Send Back for Edits returns the contract to the submitter with required comments explaining what needs to change. There is no limit on how many times a contract can bounce between reviewer and submitter.
When the submitter resubmits after edits, the contract returns to the same stage it was sent back from. The AI screening does not re-run automatically (but a reviewer can trigger a re-screen manually).
Approve advances the contract to the next stage.
Each stage transition is logged in an audit trail (who, when, action, comments).

Signature Stage:

When a contract reaches the signature stage, the system generates or retrieves the PDF version from Google Drive and sends it as an email attachment to the designated signer using Nodemailer (configure SMTP via environment variables, or use the Gmail API with the service account).
The email includes a link back to the contract in the app where the signer can confirm receipt.
After the signer signs externally, the status moves to "Awaiting Executed Upload."

Executed Upload Stage:

The submitter (or any authorized user) uploads the fully executed (signed by all parties) document.
The app stores the executed version in Google Drive alongside the original, in the same folder, with a clear naming convention: {Contract Name} - EXECUTED.pdf.
Email reminders fire on a configurable schedule (e.g., every 3 days) until the executed version is uploaded. After 30 days with no upload, escalate the reminder to the submitter's manager or the admin.
Once uploaded, status changes to "Fully Executed" and the obligation tracking engine activates.


OBLIGATION & MILESTONE TRACKING (Feature #2)
After a contract reaches "Fully Executed" status, the obligation tracking engine activates.
Obligation Types (admin-configurable, can be toggled on/off per contract type):
Obligation TypeDescriptionDefault Reminder ScheduleRenewal / ExpirationContract expiration or renewal date90, 60, 30, 14, 7 days beforePayment MilestonesScheduled payment due dates14, 7, 1 days before; 1, 7 days after (overdue)Notice PeriodsDeadline to provide termination or non-renewal notice120, 90, 60 days before the notice deadlinePerformance ReviewScheduled review dates for vendor/contractor performance14, 7 days beforeInsurance / Compliance CertificatesExpiration of required certificates60, 30, 14 days beforeRevenue RecognitionMilestones for recognizing revenue (sell-side)14, 7 days before
How it works:

When a contract becomes Fully Executed, the system prompts the user to confirm or enter obligation dates. Pre-populate from the form data where possible (expiration date, payment terms).
The admin configures which obligation types are active per contract type. For example, NDAs might only have Renewal/Expiration and Notice Period, while Vendor Agreements have all six.
A scheduled cron job runs daily, checks all obligations, and sends email reminders to the contract's submitter and any designated watchers.
Obligations appear on the dashboard widget and on the contract detail page as a timeline.
Users can mark obligations as "Completed" or "Acknowledged." Overdue obligations surface prominently on the dashboard.
The admin can configure the reminder schedule (which intervals) per obligation type.


ADMIN PANEL
The admin panel is accessible only to users with the Admin role. It includes:

Contract Types Management — CRUD for contract types. Each type has a name, description, and a JSON form schema defining the fields for that type's submission form. Provide a form builder UI (drag-and-drop or add/remove/reorder fields) so the admin doesn't need to write JSON.
Workflow Builder — visual or list-based workflow builder. Admin creates workflows, defines matching rules (direction × department × value tier), and adds ordered stages with assigned reviewers/signers.
Approval Tiers — manage the dollar-amount tiers and the designated signer for each department × tier combination. This is a matrix: rows = departments, columns = tiers, cells = designated signer (user).
AI Screening Criteria — toggle criteria on/off, add custom criteria, edit the descriptions and thresholds. Manage the required clauses list.
Obligation Configuration — toggle obligation types on/off per contract type. Configure reminder schedules.
User Management — view all users, assign/remove roles (Submitter, Legal Reviewer, Designated Signer, Admin). Display user's department from Google Workspace. Allow manual department override.
Dashboard Configuration — choose which widgets appear on the default dashboard, set thresholds (e.g., "flag contracts sitting at a stage for more than 7 days").
Company Information — manage the company's legal name, address, and other details used by the AI screening to verify contracts.
Audit Log Viewer — searchable, filterable log of all contract actions (submissions, approvals, rejections, comments, status changes, AI screening results).
